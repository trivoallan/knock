# Docs site — Docusaurus

**Status:** designed (not yet implemented)
**Date:** 2026-06-16

## Problem

houba's user documentation is now complete and organized by Diátaxis under `docs/`
(landing, tutorials, how-to, reference, explanation), but it is only browsable as raw
Markdown on GitHub. The roadmap's remaining adoption item is a **published documentation
site** — searchable, navigable, refreshed on every merge to `main`. A mandated front door
only gets adopted if users can self-serve.

## Decision

Build the site with **Docusaurus**, published to **GitHub Pages** on every merge to `main`.

Docusaurus was chosen over mkdocs-material despite the latter being the lighter,
Python-native fit, because **versioned docs and i18n are wanted at medium term** and are
first-class, natively-co-designed features of Docusaurus. The cost — a Node toolchain
alongside the Python project — is accepted and confined to a single directory.

## Design

### 1. Repo layout & content boundary

- The Docusaurus app lives in an isolated **`website/`** directory (its own
  `package.json`, `docusaurus.config.ts`, `sidebars.ts`). The Node toolchain is confined
  there; the existing Python/uv jobs are untouched.
- The `docs` plugin **sources content in place** from the repo's existing `docs/`
  (`path: '../docs'`). Content is **not** moved — the Diátaxis layout, the runnable
  `examples/*.yml` fixtures, and the paths wired into the Makefile / CI / Argo
  (`reconcile docs/examples/reference/...`) all stay put.
- **Site content (published):** `docs/index.md` (landing), `tutorials/`, `how-to/`,
  `reference/`, `explanation/`, and the `examples/README.md` catalog page.
- **Excluded from the build:** `docs/architecture/` (contributor docs — ADRs + C4),
  `docs/superpowers/` (specs/plans), and all non-`.md` files (the `.yml` fixtures, `.mmd`,
  `.dsl`). Exclusion is explicit via the plugin's `exclude` globs.

### 2. Markdown handling (the load-bearing detail)

- Set `markdown.format: 'detect'` so **`.md` files render as CommonMark, not MDX**. This
  is what lets the **generated reference** (`docs/reference/*.md` from
  `json-schema-for-humans`, dense with `{`, `<`, and HTML) render without MDX parse
  errors. MDX would choke on it.
- Pages that genuinely need JSX/React components opt in by using the **`.mdx`** extension.
  None are required for the initial content.
- Enable the **Mermaid** theme (`@docusaurus/theme-mermaid`, `markdown.mermaid: true`) for
  future use. Not load-bearing now: the C4 Mermaid exports live under the excluded
  `architecture/`.

### 3. Navigation — docs-only mode

- **Docs-only mode** (`routeBasePath: '/'`): no React splash/homepage. `docs/index.md`
  (the *Why houba* pitch) becomes the site root. No content-side React to author.
- The sidebar is **autogenerated** from the Diátaxis folder structure.
- Top navbar: Tutorials · How-to · Reference · Explanation, plus a GitHub repo link.

### 4. Cross-links to excluded content

The small set of links pointing from published pages into excluded content resolves to
**absolute GitHub URLs**:

- `explanation/index.md` → `architecture/design.md`, `architecture/decisions/`.
- `examples/README.md` catalog → the runnable `*.yml` fixtures and `scan*/README.md`.

These are contributor / runnable material whose natural home is the repo, not the site.
(Implementation note: prefer a single configured GitHub base URL over hand-edited
absolute links where practical, so the links survive a repo move.)

### 5. Build & deploy

- A dedicated **GitHub Actions** job builds the site (`npm ci && npm run build` in
  `website/`) and deploys to **GitHub Pages** on merge to `main`.
- `baseUrl: '/houba/'`, `url: 'https://trivoallan.github.io'` (project Pages site).
- The generated reference is produced by the existing `make reference` (unchanged);
  Docusaurus consumes the committed Markdown. The build does **not** regenerate it.
- The Node toolchain and this job are additive; the Python `lint` / `test` / `manifests` /
  `docker build` jobs are unchanged.

### 6. Versioning & i18n — configured-ready, not activated

Versioning and i18n are Docusaurus built-ins that require **no refactor to enable later**
(`docusaurus docs:version <v>`; an `i18n/` tree when translation begins). This increment
ships the **current docs only**. Versioning is activated at the first real need (when a
second version must be distinguished — not before, to avoid a meaningless `0.x` snapshot).
i18n is deferred until translation actually starts (the standing rule is English-only docs).

### 7. Guard-rails

- Docusaurus **fails the build on broken internal links** (`onBrokenLinks: 'throw'`). The
  Pages deploy job therefore doubles as a link-integrity check — valuable given the link
  churn during the Diátaxis reorg.
- The build job runs on pull requests too (build only, no deploy), so broken links are
  caught before merge.

## Non-goals

- No content rewriting — the Diátaxis reorg (specs of 2026-06-16, PRs #112/#113) is done.
- No React homepage / custom theme components in this increment.
- No activation of versioning or i18n (see §6).
- No change to the Python CLI, its tests, or the C4 model: the docs site is a publication
  pipeline, orthogonal to houba's runtime architecture, so the C4 model is not affected.

## Verification

- `npm run build` in `website/` succeeds locally and in CI (no broken links).
- The published site serves the landing, the four Diátaxis sections, the catalog, and the
  generated reference, with working search and sidebar.
- Links into excluded content (architecture, `.yml` fixtures) resolve to GitHub.
- The existing Python CI jobs and `make reference` are unchanged and still green.
