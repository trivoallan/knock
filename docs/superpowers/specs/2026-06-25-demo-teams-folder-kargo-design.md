## Problem statement

Show how a team uses a *folder* of MirrorPolicy files and how that folder integrates with
kargo. Today the demo runs a single reference policy (`docs/examples/reference/`) feeding one
kargo pipeline (Warehouse → dev → prod, gated by `houba verify`). The org reality houba targets
is many policies owned by different teams; the demo should make that pattern — and its payoff at
the promotion gate — visible.

## What makes this cool

**Per-team isolation of outcomes at the gate.** Team A's clean image is **promoted** while, at
the same time, Team B's vulnerable image is **held by Team B's own gate** — two independent kargo
pipelines, each owned by a different team, each gating on its own scan verdict, each traceable to
its owner via the `owners` field and the existing blast-radius owner join. The drama is the
contrast: same mechanism, opposite outcomes, attributed to the right team. It reuses the XZ
incident (`debian-xz`) already in the demo, so the payoff costs almost no new fixture.

## Constraints

- **No houba core code.** The `use_cases/loader` already walks a directory of policies, the
  `owners` field already exists on `Defaults`/`ImportProfile`, and multiple policies already place
  to distinct repos. Two teams = two MirrorPolicy files + a kargo Warehouse/Stage set per team
  (manifests) + Makefile wiring. The existing scan-attach job is reused, pointed at the teams'
  repos via its repo-list config (not new code). Nothing under `houba/`.
- **Additive, off the critical path.** `make demo` is already heavy (cert-manager, kargo, kyverno,
  argo-rollouts, ESO/OpenBao, Dependency-Track, buildkitd, registry). The two-team story is a
  dedicated target/beat, not an enlargement of the green `make demo` path.
- **Reuse incident assets.** Team B's held image is the existing `debian-xz` (CVE-2024-3094);
  Team A's promoted image is a clean base (debian/busybox). No new fixture.
- **Lean on the kargo gate, not Kyverno.** The kargo gate verifies cosign v3 attestations and
  works (validated 2026-06-25). The Kyverno admission gate is deny-only on this key-mode/kind
  setup (documented cosign-v3 limitation) — the per-team contrast is a kargo *promotion* story,
  not a Kyverno *admission* story.

## Premises (agreed in session)

1. The per-team contrast plays out at the **kargo** gate; the demo does not rely on Kyverno
   admission for this payoff.
2. Zero houba core code — loader/`owners`/multi-repo placement already support it; two teams are
   two policy files + two kargo pipeline manifests + Makefile.
3. Reuse the existing `debian-xz` incident for Team B's held image; Team A uses a clean base.
4. The "team folder" is a real directory of MirrorPolicy files, each with a distinct `owners`
   (Backstage entity-ref) and distinct destination repos, so two Warehouses watch distinct repo
   prefixes and blast-radius splits by owner.
5. The second pipeline is additive and off the critical green `make demo` path (its own
   target/beat), not a bloat of `make demo`.

## Approaches considered

### Approach A: dedicated `make demo-teams`, two policies + two kargo pipelines (minimal viable) — CHOSEN
A `docs/examples/teams/` folder with two MirrorPolicy files (each with `owners` + distinct
destination repos), a kargo "teams" component (two Warehouses + two dev→prod Stages, each gated
by `houba verify`), and an additive `make demo-teams` target that applies the team policies,
reconciles, seeds the XZ image into Team B's repo, fires both pipelines, and asserts "Team A
promoted / Team B held". Reuses the gate + incident. Off the critical path.
Effort: M · Risk: Low.

### Approach B: the two-team folder becomes the demo's default structure (ideal architecture)
Restructure the reference policy into `teams/` and have `make demo` itself run two teams.
Best "this is how an org runs houba" narrative, single demo path — but conflicts with premise 5
(bloats the heavy critical path) and forces a full e2e re-test.
Effort: L · Risk: Med.

### Approach C: kargo pipelines generated from the folder (creative/lateral)
A `deploy/scripts/policies-to-kargo.py` reads each policy's `owners`/`destinations` and generates
a Warehouse/Stage per team — "drop a policy in the folder → a pipeline appears." Shows
folder→pipeline automation and scales to N teams, but adds a generator to maintain and can
obscure the mechanism.
Effort: M/L · Risk: Med.

## Recommended approach

**Approach A.** It satisfies all five premises (additive, off the critical path, zero core,
reuses the incident) and delivers the D3 contrast payoff. C's folder→pipeline automation is
attractive but adds a generator to own; B tells the org story best but violates premise 5 on an
already-heavy demo. Start with A; C can be a follow-up once the two-team pattern proves out.

## Resolved design details (forced by spec review)

These were flagged by the adversarial review as builder blockers and are now decided:

- **Names must not collide with the existing reference pipeline.** The reference kargo component
  uses Warehouse `houba-demo` and Stages `dev`/`prod`. The teams pipelines are a **separate**,
  additive component with distinct names: Warehouses `platform` and `data`; Stages
  `platform-dev`/`platform-prod` and `data-dev`/`data-prod`. They coexist with the reference
  pipeline (the teams component is applied only by `make demo-teams`).
- **`owners` is required on the demo team policies** (the schema allows it to be `None`; the
  example files must set it). Format is the Backstage entity-ref string already used by
  `busybox.yml`: `owners: ["group:default/platform"]` for Team A, `["group:default/data"]` for
  Team B. This is what the blast-radius owner join keys on.
- **Both teams get fresh, symmetric policy files** under `docs/examples/teams/` (Team A reuses the
  *shape* of `busybox.yml`/the debian rebuild but with its own `owners` + `platform/*`
  destination; Team B places the xz image to `data/*`). Symmetric files keep the contrast legible;
  no editing of the existing reference example.
- **One kargo Project (`houba`), two pipelines.** Distinct Warehouse/Stage names (above) make one
  Project sufficient; a Project-per-team is deferred unless cross-team isolation becomes the
  teaching point.
- **Per-team `imageFrom(...)`.** Each team's `*-prod` Stage hardcodes its own
  `imageFrom(repoURL).RepoURL + "@" + ...Digest` expression for its repo (the reference
  `stage-prod.yaml` is repo-specific, so the Stage files are forked per team — the
  `houba-scan-gate` AnalysisTemplate is the only reused piece, since it takes `image-ref` as a
  parameter).
- **The gate must hold Team B for the RIGHT reason (critical CVE, not missing attestation).** So
  `make demo-teams` must run **scan-attach over both teams' repos** before firing the pipelines, so
  Team A's clean image gets a scan-pass attestation and Team B's `debian-xz` gets a
  critical-severity scan verdict. Without this, Team B's gate would hold on "no attestation",
  which is the wrong narrative.
- **Assertion is self-contained in `make demo-teams`** (not in `demo-assert-gates`), to stay off
  the critical path (premise 5). It polls kargo for each team's prod Stage: Team A's prod carries
  the promoted Freight; Team B's prod stays empty / its verification holds.

## Open questions

- **Team B's placement path:** does Team B's policy *rebuild* the vulnerable `upstream/debian-xz`
  (so the placed `data/debian-xz` still carries the bad xz, by design, to trigger the gate), or
  copy it as-is? Rebuild matches houba's front-door story but must not "fix" the xz; copy is
  simpler. Lean copy for the demo (the point is the gate catching it, not the rebuild).
- **kargo CLI vs. API for the assertion poll:** whether `make demo-teams` shells the `kargo` CLI
  or queries the Stage CRs with `kubectl` for the promoted/held check. `kubectl` on the Stage CRs
  avoids a new CLI dependency.

## Success criteria

- `make demo-teams` runs on the existing demo cluster and shows, attributable to each owner:
  Team A's clean image **promoted** to prod; Team B's `debian-xz` image **held** at its gate.
- `make demo` (the critical green path) is unchanged — the two-team story is additive.
- No change under `houba/` — only `docs/examples/teams/`, a kargo teams component, and Makefile.
- The blast-radius/owner view distinguishes the two teams.

## Distribution plan

No new artifact to distribute — this ships as repo content (example policies, deploy manifests,
a Makefile target) and docs. It rides the existing repo + `make` entry points; the Diátaxis docs
(an explanation of the team-folder/ownership pattern + a how-to for `make demo-teams`) land with
the example, per the project's examples-stay-in-sync-with-specs rule.

## Next steps

1. Author `docs/examples/teams/team-platform.yaml` (`owners: ["group:default/platform"]`, places a
   clean debian/busybox image to `platform/*`) and `team-data.yaml`
   (`owners: ["group:default/data"]`, places the vulnerable `debian-xz` to `data/*`). Both files
   set `owners` explicitly.
2. Add a separate kargo "teams" component (e.g. `deploy/components/kargo-teams/`): Warehouses
   `platform` (watches `platform/debian`) and `data` (watches `data/debian-xz`), and Stage pairs
   `platform-dev`/`platform-prod` and `data-dev`/`data-prod` — distinct names so they coexist with
   the reference pipeline. Each `*-prod` Stage carries its own `imageFrom(...)` for its repo and
   references the reused `houba-scan-gate` AnalysisTemplate.
3. Add `make demo-teams` (additive, off the critical path):
   a. ensure `seed-incident` has run (provides `upstream/debian-xz`);
   b. apply the two team policies and reconcile (places `platform/*` clean, `data/debian-xz`
      vulnerable);
   c. **run scan-attach over `platform/*` and `data/*`** so Team A gets a scan-pass attestation and
      Team B gets a critical scan verdict (this is what makes Team B's gate hold for the right
      reason);
   d. apply the kargo-teams component and let both Warehouses discover Freight;
   e. poll the Stage CRs with `kubectl`: assert `platform-prod` carries promoted Freight and
      `data-prod` is held; print the owner of each (`group:default/platform` vs `group:default/data`).
4. Wire the Diátaxis docs (explanation of the team-folder/ownership→pipeline pattern + a how-to for
   `make demo-teams`) and keep `docs/examples/` in sync.
5. (Later, optional) Approach C: generate the kargo-teams manifests from the folder.
